# 커밋 메시지 규약 (Commit Message Convention)

본 프로젝트의 Git 히스토리 가독성과 일관성을 유지하기 위해 아래의 커밋 메시지 규약을 준수합니다.

## 1. 커밋 메시지 구조

```text
<type>(<scope>): <subject>

<body>

<footer>
```

## 2. 커밋 타입 (Type)

| 타입       | 설명                                                         |
| :--------- | :----------------------------------------------------------- |
| `feat`     | 새로운 기능 추가                                             |
| `fix`      | 버그 수정                                                    |
| `docs`     | 문서 수정 (README.md, docs/\* 등)                            |
| `style`    | 코드 의미 변화 없는 스타일 수정 (세미콜론, 포맷팅 등)        |
| `refactor` | 코드 리팩토링 (기능 변화 없음)                               |
| `test`     | 테스트 코드 추가 및 수정                                     |
| `chore`    | 빌드 업무, 패키지 매니저 설정, 환경 변수 등 (기능 변화 없음) |

## 3. 작성 가이드라인

- **Subject (제목)**:
  - 50자 이내로 간결하게 작성
  - 마침표(.)를 사용하지 않음
  - 첫 글자는 영문의 경우 대문자로, 국문의 경우 명사형으로 종결 (예: "로그인 기능 구현")
- **Body (본문)**:
  - "무엇을", "왜" 변경했는지 상세히 설명 (선택 사항)
  - 한 줄당 72자 이내 권장
- **Footer (꼬리말)**:
  - 이슈 번호 참조 시 사용 (예: `Fixes: #123`, `Related to: #456`)

## 4. 커밋 예시

- `feat(reservation): 예약 승인 로직 구현`
- `fix(mariadb): DB 연결 타임아웃 오류 수정`
- `docs: 도커 설정 관련 문서 업데이트`
